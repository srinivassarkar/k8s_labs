# Kubernetes Admission Controllers

---

## The Core Idea (Remember This Always)

**Three steps for every request:**

1. **Authentication** = Who are you?
2. **Authorization** = What can you do?
3. **Admission** = Under what conditions is this allowed?

**The full request flow:**

```
kubectl → API Server → AuthN → AuthZ → Admission → etcd
                          ↓
               [Mutating Phase]
               [Validating Phase]
```

**Two important truths:**

1. **Admission controllers only work on writes** – `CREATE`, `UPDATE`, `DELETE`. They never touch reads like `GET`, `LIST`, `WATCH`.
2. **Mutating always runs before Validating** – you change the object first, then check the changed version. This order is fixed and cannot be changed.

**What Admission Does vs What Came Before:**

```
AuthN → "I know who you are"
AuthZ → "I know you're allowed to do this"
Admission → "I know the object you're submitting meets our standards"
```

- AuthN and AuthZ are about the **person or machine** making the request.
- Admission is about the **object** being created or changed.

---

## What Admission Controllers Are and Are Not

### What They Are

- Admission controllers are **plugins that intercept API requests** after auth but before the object is saved to etcd.
- They can **modify** (mutate), **validate**, or **reject** the request.
- They can see the **full object** – Pod specs, container images, labels, security settings, resource limits, and everything else.

### What They Are NOT

| **Misconception** | **Reality** |
|---|---|
| "Admission controllers control who can access the cluster" | No – that's Authentication + Authorization. |
| "Admission controllers replace RBAC" | No – they work alongside RBAC, not instead of it. |
| "Admission controllers fire on `kubectl get`" | No – reads are not checked by admission. |
| "Validating controllers run before mutating" | No – mutating always runs first. |
| "Admission rejection means the object is stored but blocked" | No – rejection means nothing is saved to etcd. |

### The Key Difference

```
Webhook Authorization → sees: user, verb, resource, namespace (NO object body)
Admission Controller → sees: full object body (Pod spec, images, labels, etc.)
```

> **Important for interviews:** If your policy needs to inspect **what** is being deployed, use admission. If it needs to evaluate **who** is making the request, use authorization.

## When to Use What – Decision Guide

### Choosing the Right Enforcement Layer

```
What does the policy need to check?
│
├── WHO is making the request?
│   (user identity, group, time of day, VPN status)
│   └── Use Webhook Authorization (e.g., OPA as authorizer)
│
└── WHAT is being deployed?
    (image name, labels, security context, resource limits)
    │
    ├── Do you need to MODIFY the object?
    │   └── Use Mutating Admission Controller
    │
    └── Do you need to ALLOW or DENY based on content?
        └── Use Validating Admission Controller
```

### Built-in vs Custom – When to Use Each

| **Need** | **Use** |
|---|---|
| Default StorageClass on PVCs | `DefaultStorageClass` (built-in) |
| Default tolerations on pods | `DefaultTolerationSeconds` (built-in) |
| Pod security standards enforced | `PodSecurity` (built-in, opt-in) |
| Namespace resource quotas | `ResourceQuota` (built-in) |
| Block privileged containers with custom logic | `ValidatingAdmissionWebhook` + Kyverno or Gatekeeper |
| Auto-inject sidecars (like Istio) | `MutatingAdmissionWebhook` + Istio or Kyverno |
| Enforce org-wide labels | `MutatingAdmissionWebhook` + Kyverno |
| Rego-based compliance with audit trails | OPA + Gatekeeper |

### OPA as Authorizer vs OPA as Admission Controller

| **Dimension** | **Webhook Authorization** | **Admission Control** |
|---|---|---|
| When it runs | After AuthN, before Admission | After AuthZ, before etcd write |
| What it sees | User, verb, resource, namespace | Full object body |
| Can inspect image field? | ❌ No | ✅ Yes |
| Can mutate the object? | ❌ No | ✅ Yes (mutating only) |
| How common in production | Rare – only for high-security orgs | Very common – most production clusters |
| Example use cases | Time-based access, VPN checks, HR integration | Image registry rules, label enforcement, security contexts |

### Kyverno vs Gatekeeper – Which to Choose

| **Dimension** | **OPA + Gatekeeper** | **Kyverno** |
|---|---|---|
| Policy language | Rego (a new language to learn) | Kubernetes-native YAML |
| Validation | ✅ Excellent | ✅ Excellent |
| Mutation | ⚠️ Limited (only via Mutation CRDs) | ✅ Powerful (JSON patch, conditional) |
| Resource generation | ❌ Not supported | ✅ Supported |
| Learning curve | High (Rego) | Low (YAML) |
| Image signature verification | ❌ No | ✅ Yes |
| Best for | Compliance-heavy, enterprise, teams with Rego expertise | DevOps teams, fast-moving teams, automation |

> **Rule of thumb:** Use Gatekeeper when your security team needs complex policies and audit trails. Use Kyverno when your platform team needs to move fast with YAML and needs mutation + generation + validation in one tool.

## How Admission Actually Works

### The Full Admission Flow (Step by Step)

```
1. Request arrives at API Server (kubectl apply, API call, controller)
2. Authentication – who is this?
3. Authorization – are they allowed to do this?
4. ─── ADMISSION PHASE BEGINS ───────────────────────────────────
5. MUTATING PHASE:
   a. All mutating admission plugins run IN ORDER.
   b. Each plugin receives the object, may change it, returns the patched object.
   c. The object is changed in memory – NOT saved to etcd yet.
   d. If a webhook is configured, API Server sends an HTTP POST to the webhook server.
   e. Webhook returns: allowed + optional JSON patch.
6. Object now holds the final mutated form.
7. VALIDATING PHASE:
   a. All validating admission plugins run IN ORDER.
   b. Each plugin receives the (now mutated) object.
   c. Each plugin returns: allowed: true or allowed: false + reason.
   d. If ANY validating plugin returns false → request DENIED immediately.
   e. User gets an error, nothing is written to etcd.
8. All validating plugins pass → object written to etcd.
9. Response returned to caller.
```

### How Dual-Function Controllers Work

Some controllers like `LimitRanger` and `ServiceAccount` do both mutation and validation. They are internally split:

```
Mutating Phase → LimitRanger sets default CPU/memory
                 ServiceAccount assigns default SA
Validating Phase → LimitRanger checks limits are within allowed range
                   ServiceAccount verifies SA exists and is authorized
```

The plugin appears once in the `--enable-admission-plugins` list, but its logic runs in both phases.

### Webhook Execution Detail

When a `MutatingAdmissionWebhook` or `ValidatingAdmissionWebhook` fires:

```
1. API Server matches the request against WebhookConfiguration rules.
2. API Server sends an AdmissionReview object (JSON) to the webhook server via HTTPS.
3. Webhook server processes the request.
4. Webhook server returns an AdmissionReview response:
   {
     "allowed": true/false,
     "status": { "message": "reason if denied" },
     "patch": [...],   ← mutating only
     "patchType": "JSONPatch"   ← mutating only
   }
5. API Server applies the patch (if mutating) and continues.
```

### Execution Order Reality

Here's an example of `--enable-admission-plugins`:

```bash
--enable-admission-plugins=NamespaceLifecycle,LimitRanger,ServiceAccount,\
MutatingAdmissionWebhook,ValidatingAdmissionWebhook,ResourceQuota
```

**The actual order of execution:**

```
MUTATING PHASE (order from plugin list):
1. NamespaceLifecycle (no mutating logic – skipped)
2. LimitRanger (sets default limits)
3. ServiceAccount (assigns default SA)
4. MutatingAdmissionWebhook (sidecar injection, label defaults)

VALIDATING PHASE (order from plugin list):
1. NamespaceLifecycle (validates namespace exists and is active)
2. LimitRanger (validates limits within bounds)
3. ServiceAccount (validates SA exists)
4. ValidatingAdmissionWebhook (OPA/Kyverno/Gatekeeper checks)
5. ResourceQuota (enforces namespace resource caps)
```

> **Key insight:** The order in `--enable-admission-plugins` is not "mutating then validating" – it's the order within each phase. Kubernetes internally separates which phase each plugin runs in.

### What Mutating Controllers Can and Cannot Do

```
CAN: Add labels, annotations
CAN: Set default values (image pull policy, tolerations)
CAN: Inject containers (sidecars)
CAN: Modify resource limits
CANNOT: Deny the request – only validating controllers can deny
```

> **This is an interview trap.** Mutating admission controllers **cannot reject requests**. They can only transform. Rejection is only for validating controllers.

## Hands-On Examples – YAML + Commands

### Enable/Disable Admission Plugins

```bash
# For kubeadm-based clusters – edit the static pod manifest
vi /etc/kubernetes/manifests/kube-apiserver.yaml

# Add/modify this flag under spec.containers[0].command:
- --enable-admission-plugins=NodeRestriction,PodSecurity,ResourceQuota
- --disable-admission-plugins=DefaultStorageClass
```

```bash
# Check what's currently enabled (systemd-based)
ps -aux | grep kube-apiserver | grep admission

# On kubeadm/KIND
cat /etc/kubernetes/manifests/kube-apiserver.yaml | grep admission
```

### ValidatingWebhookConfiguration

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: my-validating-webhook
webhooks:
- name: validate.myorg.com
  admissionReviewVersions: ["v1"]
  clientConfig:
    service:
      name: my-webhook-service
      namespace: webhook-system
      path: /validate
    caBundle: <base64-encoded-CA-cert>   # must match webhook server TLS cert
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["pods"]
    scope: "Namespaced"
  failurePolicy: Fail   # Fail = deny if webhook unreachable; Ignore = allow
  sideEffects: None
  namespaceSelector:
    matchLabels:
      admission: enabled   # only fire in namespaces with this label
```

### MutatingWebhookConfiguration

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: my-mutating-webhook
webhooks:
- name: mutate.myorg.com
  admissionReviewVersions: ["v1"]
  clientConfig:
    service:
      name: my-webhook-service
      namespace: webhook-system
      path: /mutate
    caBundle: <base64-encoded-CA-cert>
  rules:
  - apiGroups: ["apps"]
    apiVersions: ["v1"]
    operations: ["CREATE"]
    resources: ["deployments"]
    scope: "Namespaced"
  failurePolicy: Ignore   # Ignore = allow even if webhook is down
  sideEffects: None
  reinvocationPolicy: Never   # Never or IfNeeded (for chained mutations)
```

### PodSecurity Admission (Built-in, opt-in)

```yaml
# Label a namespace to enforce pod security standards
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted   # enforce – blocks violating pods
    pod-security.kubernetes.io/warn: baseline        # warn – allows but warns
    pod-security.kubernetes.io/audit: baseline       # audit – logs violations
```

**Security levels:**
- `privileged` – no restrictions
- `baseline` – blocks known privilege escalations
- `restricted` – hardened, follows current best practices

### ResourceQuota (Built-in Validating)

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-a
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
    count/deployments.apps: "10"
```

### LimitRange (Built-in Dual-Function)

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: team-a
spec:
  limits:
  - type: Container
    default:            # injected if not specified (mutating)
      cpu: 500m
      memory: 256Mi
    defaultRequest:     # injected if not specified (mutating)
      cpu: 100m
      memory: 128Mi
    max:                # ceiling – violation denied (validating)
      cpu: "2"
      memory: 2Gi
```

### Kyverno Policy Examples

**Validating policy – deny privileged containers:**

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-privileged-containers
spec:
  validationFailureAction: Enforce   # Enforce = deny; Audit = log only
  rules:
  - name: check-privileged
    match:
      any:
      - resources:
          kinds: [Pod]
    validate:
      message: "Privileged containers are not allowed."
      pattern:
        spec:
          containers:
          - =(securityContext):
              =(privileged): "false"
```

**Mutating policy – add labels if missing:**

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: add-default-labels
spec:
  rules:
  - name: add-team-label
    match:
      any:
      - resources:
          kinds: [Deployment]
    mutate:
      patchStrategicMerge:
        metadata:
          labels:
            +(team): "unassigned"   # "+" = only add if missing
```

### Check Which Admission Controllers Are Running

```bash
# kubeadm/KIND
kubectl -n kube-system get pod kube-apiserver-<node> -o yaml | grep admission

# List registered webhook configurations
kubectl get validatingwebhookconfigurations
kubectl get mutatingwebhookconfigurations

# Describe a webhook to see its rules and failure policy

```

## Production Setup – Real-World Architecture

### Typical Production Cluster Admission Stack

```
API Server flags:
--authorization-mode=Node,RBAC
--enable-admission-plugins=NodeRestriction,MutatingAdmissionWebhook,\
ValidatingAdmissionWebhook,PodSecurity,ResourceQuota
```

**The admission layer looks like this:**

```
┌─────────────────────────────────────────────────────────────┐
│ ADMISSION CONTROL STACK                                    │
│                                                             │
│ Built-in:                                                   │
│   NodeRestriction – kubelet cannot touch other nodes        │
│   PodSecurity – enforce baseline/restricted per namespace   │
│   ResourceQuota – cap CPU/memory per namespace              │
│   LimitRanger – default + max limits per namespace          │
│                                                             │
│ Custom (via webhooks):                                      │
│   Kyverno – label enforcement, mutation, generation         │
│   OR                                                        │
│   OPA + Gatekeeper – Rego-based compliance validation       │
└─────────────────────────────────────────────────────────────┘
```

### Webhook Server Architecture (Inside Cluster)

```
┌─────────────┐   AdmissionReview   ┌──────────────────────┐
│ API Server  │ ─────────────────────▶│ Webhook Deployment  │
│             │◀─────────────────────│ (Kyverno/Gatekeeper)│
│             │  AdmissionResponse   │ - TLS terminated    │
└─────────────┘                      │ - Service exposed   │
                                     │ - HA replicas      │
                                     └──────────────────────┘
```

> Webhook servers are almost always deployed **inside the cluster** as Deployments with a Service. Running them outside adds latency, TLS complexity, and availability risks.

### `failurePolicy` – The Decision That Matters Most

```
failurePolicy: Fail
→ If webhook is unreachable or times out → REQUEST DENIED
→ Use for: security-critical webhooks (registry checks, privilege blocks)
→ Risk: webhook outage blocks all matching requests

failurePolicy: Ignore
→ If webhook is unreachable or times out → REQUEST ALLOWED
→ Use for: non-critical mutations (default labels, annotations)
→ Risk: policy gaps during webhook downtime
```

> In production, security-critical webhooks (`Fail`) need **high availability** (≥2 replicas, PodDisruptionBudget, resource limits). Non-critical webhooks can use `Ignore`.

### Namespace Selector Pattern

```yaml
namespaceSelector:
  matchLabels:
    admission: enabled
```

This controls **which namespaces a webhook fires in**. Common pattern:

```
kube-system      → no admission label → webhook skips (avoids bootstrapping loops)
production       → admission: enabled → webhook fires
staging          → admission: enabled → webhook fires
dev              → (no label) → webhook skips (fast iteration)
```

> Always exclude `kube-system` and `kube-public` from custom webhooks. Accidentally blocking system pods can break the cluster.

### The Layered Policy Model (Enterprise)

```
Layer 1 – Platform team owns:
  Built-in admission plugins (PodSecurity, ResourceQuota, LimitRanger)
  These are cluster-wide, non-negotiable baselines

Layer 2 – Security team owns:
  Kyverno or Gatekeeper ClusterPolicies
  Registry restrictions, privilege escalation blocks, image signing

Layer 3 – App teams own:
  Namespace-scoped Kyverno policies
  Team-specific conventions (naming, labeling, resource sizing)
```

## Common Mistakes (What Actually Breaks)

### ❌ Mistake 1 – `failurePolicy: Fail` on a Single-Replica Webhook

**What happens:** Webhook pod crashes during deployment. Every pod create in matching namespaces returns a webhook error. Incident.

**Fix:**
```yaml
# Always run admission webhooks with HA
spec:
  replicas: 2   # minimum; 3 preferred for critical webhooks
```

```yaml
# Add PDB to prevent both replicas going down during node drain
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: webhook-pdb
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: my-webhook
```

### ❌ Mistake 2 – Webhook Fires in `kube-system`

**What happens:** Kyverno or custom webhook applies to `kube-system`. A new system pod violates a policy. System component can't start. Cluster is partially broken.

**Fix:** Always add a `namespaceSelector` that excludes `kube-system`:

```yaml
namespaceSelector:
  matchExpressions:
  - key: kubernetes.io/metadata.name
    operator: NotIn
    values: ["kube-system", "kube-public"]
```

### ❌ Mistake 3 – Validating Policy on Already-Existing Resources

**What happens:** You install a new Kyverno policy that blocks pods without a `team` label. All future pods are blocked. Existing pods without the label are unaffected – until they restart. Engineers are confused why new deployments fail but old ones work.

**Fix:** Start with `validationFailureAction: Audit` to discover violations before enforcing. Then fix existing resources, then flip to `Enforce`.

### ❌ Mistake 4 – Ordering Mistake: Validating Before Full Mutation

**What happens:** Custom webhook validates the image field before a mutating webhook has had a chance to rewrite the image tag. Validation fails even though the final image would be valid.

**Fix:** Remember that all mutating webhooks run before all validating ones. If you own both, ensure your mutating webhook runs first by ordering the webhook configurations appropriately. Use `reinvocationPolicy: IfNeeded` if needed.

### ❌ Mistake 5 – TLS Certificate Mismatch

**What happens:** Webhook server's TLS cert doesn't match the `caBundle` in `WebhookConfiguration`. API Server refuses to call the webhook. All matching requests are denied (if `failurePolicy: Fail`) or silently passed (if `Ignore`).

**Fix:**
```bash
# Verify the CA bundle in the webhook config matches the cert the server is serving
kubectl get validatingwebhookconfiguration <name> -o jsonpath='{.webhooks[0].clientConfig.caBundle}' | base64 -d | openssl x509 -text -noout
```

### ❌ Mistake 6 – ResourceQuota Without LimitRange

**What happens:** Quota enforces limits on the namespace. But pods don't specify requests/limits. Kubernetes can't count unset resources against the quota. Pods get created without resource constraints, quota appears unused, actual resource usage is uncontrolled.

**Fix:** Always pair `ResourceQuota` with `LimitRange`. LimitRange sets defaults so every container has requests/limits, which then count against the quota.

### ❌ Mistake 7 – PodSecurity Enforced on a Running Namespace

**What happens:** You add `pod-security.kubernetes.io/enforce: restricted` to a namespace that has existing pods violating the restricted profile. Existing pods are unaffected (admission only fires on new writes). New pod replicas fail. Rollout is blocked.

**Fix:** Use `warn` and `audit` labels first, run a dry-run check, fix existing workloads, then add `enforce`.

## Interview Answers (Compressed, Ready to Say)

### Q: What is an admission controller and where does it fit in the request lifecycle?

> "An admission controller is a plugin that intercepts API write requests after authentication and authorization have succeeded, but before the object is persisted to etcd. They sit in the admission phase. There are two types – mutating controllers that modify the object, and validating controllers that allow or deny it. Mutating always runs first so that validating controllers evaluate the final form of the object."

### Q: What's the difference between a mutating and a validating admission controller?

> "A mutating admission controller transforms the object – it can inject sidecars, set defaults, add labels, or patch the spec. It cannot deny the request. A validating admission controller evaluates the object and either allows or rejects it based on policy. The critical execution order is: all mutating controllers run first, then all validating controllers run on the mutated result. If any validating controller denies, the entire request is rejected and nothing is written to etcd."

### Q: What's the difference between webhook authorization and admission control?

> "Webhook authorization runs before admission and evaluates the identity and context of the request – who is making it, what verb, what resource, what namespace. It cannot see the object body. Admission control runs after authorization and can inspect the full object – the pod spec, container images, labels, security contexts. So if I need to block requests from users outside business hours, that's webhook authorization. If I need to block pods using images from untrusted registries, that's an admission controller, because only admission can see the image field."

### Q: Can a mutating admission controller deny a request?

> "No. Mutating admission controllers can only transform the object. The ability to deny belongs exclusively to validating admission controllers. This is by design – mutation is about transformation, validation is about enforcement. If you need to deny, you need a validating controller."

### Q: What is `failurePolicy` and why does it matter?

> "failurePolicy controls what happens if the webhook server is unreachable or times out. Setting it to `Fail` means the API Server treats an unreachable webhook as a denial – the request is blocked. Setting it to `Ignore` means the request is allowed through even if the webhook can't be reached. For security-critical webhooks like registry enforcement or privilege checks, you want `Fail` to ensure the policy is always enforced. But then you need the webhook itself to be highly available, otherwise it becomes a single point of failure for your cluster."

### Q: What does PodSecurity admission controller do and how does it replace PSPs?

> "PodSecurity is a built-in validating admission controller that enforces the Pod Security Standards – three profiles: privileged, baseline, and restricted. You apply it per namespace using labels, with three modes: enforce blocks violating pods, warn allows them but prints a warning, and audit allows them but logs the violation. It replaced PodSecurityPolicies which were deprecated in 1.21 and removed in 1.25. The key improvement is that PodSecurity is much simpler – you don't need RBAC bindings to make it work, you just label the namespace."

### Q: When would you choose Kyverno over OPA Gatekeeper?

> "I'd choose Kyverno for most teams because it uses Kubernetes-native YAML for all policies – validation, mutation, and resource generation – with no new language to learn. It also supports image signature verification and can generate resources like NetworkPolicies or Secrets automatically. I'd choose Gatekeeper when the team has Rego expertise and needs the expressiveness of OPA's policy language for complex validation logic, or when there's already Gatekeeper in use across the org and we need consistency. The main limitation of Gatekeeper is that mutation is constrained to specific Mutation CRDs and it has no resource generation capability."

### Q: What happens if two mutating webhooks both modify the same field?

> "The last mutation wins. Mutating webhooks run in the order they're configured. If two webhooks both patch the same field, the second one's value is what the validating phase sees. For cases where chained mutations need to be aware of each other's changes, you can set `reinvocationPolicy: IfNeeded` which causes the first webhook to be reinvoked after subsequent webhooks have run – but this can create loops if not designed carefully."

## Debugging – Fast Diagnosis Paths

### Request Denied at Admission – Diagnosis Path

```
API call returns error
│
├── Error says "admission webhook ... denied the request"
│   │
│   ├── Check which webhook denied it
│   │   kubectl get validatingwebhookconfigurations
│   │   kubectl describe validatingwebhookconfiguration <name>
│   │
│   └── Check webhook server logs
│       kubectl logs -n <webhook-ns> -l app=<webhook> --tail=50
│
├── Error says "webhook call failed" or "connection refused"
│   │
│   ├── Webhook server is down
│   │   kubectl get pods -n <webhook-ns>
│   │
│   ├── Check failurePolicy
│   │   If Fail → request blocked until webhook is up
│   │   If Ignore → request should pass – something else is wrong
│   │
│   └── Check TLS cert mismatch
│       kubectl get validatingwebhookconfiguration <name> -o yaml
│       Verify caBundle matches webhook server cert
│
└── Error is from a built-in controller (LimitRanger, ResourceQuota, etc.)
    │
    └── kubectl describe resourcequota -n <namespace>
        kubectl describe limitrange -n <namespace>
        kubectl describe namespace <namespace>
```

## 30-Second Quick Revision

### The Essentials

```
Admission = checks the object, not the user (that's AuthZ)
Only fires on writes – CREATE, UPDATE, DELETE (not GET/LIST/WATCH)
```

### Two Types – Order Matters

```
Mutating   → transforms object, CANNOT deny
Validating → evaluates object, CAN deny
Order: Mutating FIRST, then Validating – always
```

### Built-in Key Plugins (Remember These)

```
DefaultStorageClass – PVC gets default SC
LimitRanger – default + max limits (mutates + validates)
ResourceQuota – namespace resource cap (validates)
PodSecurity – enforce/warn/audit per namespace (opt-in)
NodeRestriction – kubelet can't touch other nodes
ServiceAccount – assigns default SA (mutates + validates)
MutatingAdmissionWebhook – bridge to custom mutating webhooks
ValidatingAdmissionWebhook – bridge to custom validating webhooks
```

### Custom Tools

```
Kyverno         → YAML, validate + mutate + generate, image signing
OPA+Gatekeeper  → Rego, validate + limited mutate, no generate
```

### `failurePolicy` – Quick Decision

```
Fail   → webhook down = request denied (secure but risky)
Ignore → webhook down = request passes (less secure)
```

### Golden Rules

```
Never webhook into kube-system
Always pair ResourceQuota with LimitRange
Mutating webhook cannot deny – only validating can
Use Audit mode first before Enforce for new policies
```

### Pod Security Standards Levels

| **Level** | **What It Blocks** |
|---|---|
| `privileged` | Nothing – no restrictions |
| `baseline` | Known privilege escalations (hostPID, hostNetwork, privileged containers) |
| `restricted` | All of baseline + must run as non-root, read-only root FS recommended, no capabilities |

### Namespace PodSecurity Labels

```bash
# Enforce restricted, warn on baseline
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=baseline \
  pod-security.kubernetes.io/audit=baseline

  pod-security.kubernetes.io/enforce=restricted \
```
