# Kubernetes Authentication & Service Accounts


---

## The Core Idea (Remember This Always)

**Two simple steps for every request:**
1. **AUTHENTICATION** = Who are you?
2. **AUTHORIZATION** = What can you do?

**The Golden Rule:**
> **Every request goes through the API Server first. Nothing bypasses it. Ever.**

```
Request → API Server → Check WHO → Check WHAT → Allow or Deny
```

---

## What Kubernetes Does vs What It Doesn't

| **Kubernetes Handles** | **Kubernetes Does NOT Handle** |
|---|---|
| Checks who is making the request | Manage human user accounts |
| Decides what they can do | Store users in a database |
| Issues tokens for machines (pods, tools) | Act as a login system for people |

**Key Point:**
- **Humans** → Identity comes from outside (certificates, Google login, company login)
- **Machines** → Identity comes from ServiceAccounts (managed by Kubernetes)

> **There is no `kubectl create user` command. Human users don't exist inside Kubernetes.**

## Authentication Methods Made Simple

| **Method** | **Who Uses It** | **Should You Use?** |
|---|---|---|
| **Certificates** | Humans | ✅ Yes - Common for admins |
| **OIDC/LDAP** | Humans | ✅ Yes - Company login (Google, Okta) |
| **Projected Tokens** | Pods (inside cluster) | ✅ Yes - Auto-mounted, auto-rotated |
| **TokenRequest API** | External tools (Jenkins) | ✅ Yes - Short-lived, on-demand |
| **Secret-based Tokens** | Any machine | ❌ No - Old method, long-lived, unsafe |
| **Basic Auth** | Humans | ❌ Dead - Never use |
| **Static Token File** | Humans | ❌ Dead - Never use |

## Simple Decision Flowchart

```
Who needs access?
│
├── HUMAN
│   └── Use certificates OR company login (OIDC)
│
└── MACHINE
    │
    ├── Inside Cluster? (Pod)
    │   └── Use ServiceAccount + Projected Token (auto-mounted)
    │
    └── Outside Cluster? (Jenkins, GitHub Actions)
        └── Use TokenRequest API (short-lived token)
```

### Quick Token Type Guide

| **Situation** | **Token Type** | **Why** |
|---|---|---|
| Pod needs API access | Projected token | Auto-mounted, auto-rotated |
| Jenkins/GitHub Actions | TokenRequest API | Short-lived, manual refresh needed |
| Old clusters (before v1.24) | Secret-based token | Only for backward compatibility |
| High-security companies | OIDC + Workload Identity | No tokens stored anywhere |

> **Warning:** Never give `cluster-admin` to a pod or CI system. Never use the `default` ServiceAccount for anything important. Never use long-lived tokens when short-lived ones exist.

## How Authentication Actually Works

### What Happens Inside a Pod

When a pod starts, Kubernetes automatically puts a token here:

```
/var/run/secrets/kubernetes.io/serviceaccount/
├── token        ← The credential (JWT)
├── ca.crt       ← Certificate to verify API server
└── namespace    ← Which namespace the pod is in
```

### Step-by-Step Flow

```
1. Pod starts
2. Kubelet mounts the token into the pod
3. Pod makes a request to API Server with:
   Authorization: Bearer <token>
4. API Server:
   → Checks token signature (is it valid?)
   → Reads the JWT to find the ServiceAccount name
   → Checks RBAC rules (is this allowed?)
5. Request is ALLOWED or DENIED
```

### What's Inside the Token (JWT)

```json
{
  "iss": "kubernetes/serviceaccount",
  "sub": "system:serviceaccount:jenkins:jenkins-sa",
  "aud": ["kubernetes"],
  "exp": 1700000000,
  "namespace": "jenkins"
}
```

**Important Fields:**
- **sub** = The ServiceAccount identity (used for RBAC checks)
- **aud** = Who can use this token (prevents misuse)
- **exp** = When token expires (short-lived by design)

### Groups Auto-Assigned to ServiceAccounts

When a ServiceAccount authenticates, it automatically belongs to these groups:

```
system:serviceaccount:<namespace>:<name>     ← Specific SA
system:serviceaccounts:<namespace>           ← All SAs in namespace
system:serviceaccounts                       ← All SAs in cluster
```

> **You can grant permissions to ALL ServiceAccounts in a namespace:** This gives access to every SA in that namespace. Use this carefully!

### Token Rotation (When They Auto-Update)

| **Token Type** | **Auto-Rotation?** |
|---|---|
| Projected tokens (inside pods) | ✅ Yes - Kubelet handles it |
| TokenRequest API (external) | ❌ No - You must refresh manually |
| Secret-based tokens (old) | ❌ No - Never rotates |

> **Interview Tip:** Only projected tokens inside pods auto-rotate. External tokens are your responsibility to refresh.

## Hands-On Examples

### Step 1: Create Namespace and ServiceAccount

```bash
kubectl create ns jenkins
kubectl create sa jenkins-sa -n jenkins
```

### Step 2: Create a ClusterRole (Define Permissions)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: jenkins-role
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list", "create", "delete"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "delete"]
```

### Step 3: Bind the Role to the ServiceAccount

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: jenkins-binding
subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: jenkins
roleRef:
  kind: ClusterRole
  name: jenkins-role
```

```bash
kubectl apply -f cr-crb.yaml
```

### Step 4: Generate a Token (Modern Way)

```bash
# Create a short-lived token
kubectl create token jenkins-sa -n jenkins

# With custom expiry time
kubectl create token jenkins-sa -n jenkins --duration=1h
```

### Step 5: Use the Token

```bash
TOKEN=$(kubectl create token jenkins-sa -n jenkins)
curl -H "Authorization: Bearer $TOKEN" https://<api-server>/api
```

### Step 6: Assign SA to a Pod

```yaml
spec:
  serviceAccountName: jenkins-sa
  # Token auto-mounted - no extra steps needed
```

## Old Secret-Based Tokens (Know It, But Don't Use It)

### How It Used to Work (Before v1.24)

```
1. Create ServiceAccount
2. Create a Secret pointing to the SA
3. Kubernetes generates a token and stores it in etcd
4. Extract token manually
5. Token NEVER expires ← BIG PROBLEM
```

### What This Looks Like

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: jenkins-sa-secret
  namespace: jenkins
  annotations:
    kubernetes.io/service-account.name: jenkins-sa
type: kubernetes.io/service-account-token
```

```bash
# Extract token
kubectl get secret jenkins-sa-secret -n jenkins \
  -o jsonpath="{.data.token}" | base64 --decode
```

### What Changed in v1.24

| **Before v1.24** | **After v1.24** |
|---|---|
| SA auto-creates a Secret | No auto-created Secret |
| Token stored in etcd | Token issued on-demand |
| Token never expires | Token has expiry time |
| Rotation = manual | Rotation = automatic (for pods) |

### When Would You Use Legacy Tokens?

**Only in these rare cases:**
- Clusters running Kubernetes < v1.24
- Systems that can't refresh tokens
- Quick hacks in air-gapped environments (still risky!)

> **What to Say in Interviews:** "I would never use Secret-based tokens. They don't expire and are stored in etcd, creating a permanent security risk. The TokenRequest API is the correct approach."

## Production Setup Examples

### Internal Tools (Inside Cluster)

Tools like Prometheus, ArgoCD, and Fluentd follow this pattern:

```
┌────────────────────────────────────────────┐
│           KUBERNETES CLUSTER              │
│                                           │
│  [Prometheus Pod] ←→ [API Server]        │
│     ↑ (token auto-mounted)               │
│                                           │
└────────────────────────────────────────────┘
          ↑
  [Grafana UI] talks to Prometheus
  (Never touches API Server directly)
```

**Key Point:** External UIs never talk to the API Server directly. The tool inside the cluster does, using its ServiceAccount token.

### External Tools (Outside Cluster)

Tools like Jenkins and GitHub Actions:

```
1. Jenkins requests token → kubectl create token jenkins-sa
2. API Server issues short-lived token
3. Jenkins calls API Server with token
4. Token expires after TTL (e.g., 1 hour)
5. Jenkins must request a new token
```

### Advanced Security: OIDC + Workload Identity

```
→ No tokens stored anywhere
→ Identity comes from external system (GitHub Actions OIDC)
→ Kubernetes trusts the external issuer directly
```

### The `automountServiceAccountToken` Setting

By default, every pod gets a token mounted. For pods that don't need API access, disable it:

```yaml
# On ServiceAccount (applies to all pods using it)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
automountServiceAccountToken: false

# Or on Pod spec (overrides SA setting)
spec:
  automountServiceAccountToken: false
```

> **Why This Matters:** If your app doesn't call the Kubernetes API, it shouldn't have credentials. This limits damage if the pod is compromised.

### The `aud` (Audience) Claim

Modern tokens are **audience-bound**:

```json
"aud": ["kubernetes"]
```

A token for `kubernetes` can't be used against other systems (like your internal metrics API). This prevents stolen tokens from being misused.

You can create tokens with custom audiences:

```bash
kubectl create token jenkins-sa --audience=https://my-internal-service
```

## Common Mistakes (What Breaks in Real Life)

### ❌ Mistake 1: Giving `cluster-admin` to Automation

```yaml
roleRef:
  kind: ClusterRole
  name: cluster-admin   # ← NEVER for CI/CD!
```

**Fix:** Create a custom role with only the permissions needed.

### ❌ Mistake 2: Using the `default` ServiceAccount

Every namespace has a `default` SA. Pods use it automatically. If anyone binds permissions to `default`, **every pod** in that namespace gets those permissions.

```bash
# ❌ DANGEROUS
kubectl create rolebinding ... --serviceaccount=myns:default
```

**Fix:** Always create a dedicated SA for each workload.

### ❌ Mistake 3: Hardcoding Long-Lived Tokens in CI/CD

```
Jenkins hardcodes token in pipeline
Token leaks in logs or config
Attacker has permanent cluster access
```

**Fix:** Use `kubectl create token` with short TTL. Rotate before expiry.

### ❌ Mistake 4: Token Expires Mid-Pipeline

```
Jenkins pipeline starts with valid token
Pipeline runs > 1 hour
Token expires
API calls start failing (401)
Pipeline fails
```

**Fix:** Request token with sufficient TTL or implement refresh logic.

### ❌ Mistake 5: Pod Gets 403 Errors

**Common causes:**
1. Pod using `default` SA (no permissions)
2. Forgot to create RoleBinding
3. RoleBinding in wrong namespace
4. Typo in `serviceAccountName`

### ❌ Mistake 6: Namespace Mismatch in Binding

```yaml
subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: production   # ← Must match where SA lives!
```

### ❌ Mistake 7: Role vs ClusterRole Confusion

| **Resource** | **Scope** |
|---|---|
| Role | Single namespace only |
| ClusterRole | Cluster-wide OR reusable across namespaces |
| RoleBinding | Grants within a namespace |
| ClusterRoleBinding | Grants cluster-wide |

**Important:** You **can** bind a ClusterRole with a RoleBinding (restricts to namespace). You **cannot** bind a Role with a ClusterRoleBinding.

## Interview Questions (Simple Answers)

### Q: How do pods authenticate to the Kubernetes API?

> "Pods use ServiceAccount tokens automatically mounted by the kubelet at `/var/run/secrets/kubernetes.io/serviceaccount/token`. These are short-lived JWTs issued via the TokenRequest API. The API Server validates the token and checks RBAC permissions."

### Q: What's the difference between internal and external tools?

> "Internal tools like Prometheus use projected tokens that are auto-mounted and auto-rotated. External tools like Jenkins use the TokenRequest API to get short-lived tokens and must refresh them manually."

### Q: Why use ServiceAccounts instead of user credentials?

> "ServiceAccounts give machines a distinct identity separate from humans. Audit logs show `system:serviceaccount:jenkins:jenkins-sa` performed an action - not `admin`. This enables fine-grained permissions for each workload."

### Q: What changed in Kubernetes v1.24 regarding tokens?

> "Before v1.24, every SA automatically got a Secret with a long-lived token. After v1.24, no auto-Secret is created. Tokens are issued on-demand via TokenRequest API, have expiry times, and projected tokens auto-rotate."

### Q: Would you ever use Secret-based long-lived tokens?

> "No. They don't expire, are stored in etcd, and manual rotation almost never happens. Use TokenRequest API for external tools and projected tokens for pods. For advanced setups, use OIDC federation."

### Q: What is `automountServiceAccountToken` and when do you use it?

> "It controls whether a token is automatically mounted. I disable it for any pod that doesn't need to call the Kubernetes API. This reduces attack surface - a compromised pod with no token can't pivot to the API Server."

### Q: A pod is getting 403 errors. How do you debug?

> "First check if the pod has the right `serviceAccountName`. Verify the ServiceAccount exists in the correct namespace. Check if there's a RoleBinding or ClusterRoleBinding linking that SA to a role. Use `kubectl auth can-i` to test permissions."

## Debugging (How to Fix Issues Fast)

### Check What a ServiceAccount Can Do

```bash
# Can it get pods?
kubectl auth can-i get pods \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins

# Can it create deployments?
kubectl auth can-i create deployments \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins

# Show all permissions
kubectl auth can-i --list \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins
```

### Check What's Mounted in a Pod

```bash
# Which SA is the pod using?
kubectl get pod <pod-name> -o jsonpath='{.spec.serviceAccountName}'

# See mounted volumes
kubectl describe pod <pod-name>
```

### Inspect the Token

```bash
# Read token from inside pod
kubectl exec <pod-name> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token

# Decode it (copy to jwt.io or use command)
TOKEN=$(kubectl exec <pod> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token)
echo $TOKEN | cut -d. -f2 | base64 -d | jq .
```

### Check RBAC Bindings

```bash
# List all bindings in namespace
kubectl get rolebindings -n jenkins -o wide

# Find bindings for a specific SA
kubectl get clusterrolebindings -o json | \
  jq '.items[] | select(.subjects[]?.name=="jenkins-sa")'

# Describe a specific binding
kubectl describe clusterrolebinding jenkins-binding
```

### Common 403 Diagnosis Path

```
403 Error on API call
↓
kubectl auth can-i [verb] [resource] --as=system:serviceaccount:[ns]:[sa]
↓
If "no" → Check RoleBinding exists
↓
kubectl get rolebindings,clusterrolebindings -A | grep [sa-name]
↓
If binding exists → Check Role has correct verbs/resources
↓
kubectl describe clusterrole [role-name]
```

```
Token = Proof of Identity
RBAC = Permission Check

```

### After v1.24 (Important)

```
No auto-Secret created
No long-lived tokens by default
Use TokenRequest API always
```

```
automountServiceAccountToken: false → Reduce attack surface
```
