---

```bash
--basic-auth-file=/etc/kubernetes/auth.csv
```

❌ Plain text

❌ No rotation

❌ Dead in production

```bash

```

❌ Same problems

❌ Not used in real-world

✔ kubeconfig

✔ signed by CA

✔ used by kubectl

✔ OIDC (Google, Azure, Okta)

✔ LDAP / AD

👉 Enables:

* SSO
* MFA
* central identity

# 🧠 4. Service Account Mental Model

```
ServiceAccount = identity for machines
```

---

### Why needed?

Instead of:

```
User: admin created pod ❌
```

We want:

```
ServiceAccount: jenkins-deployer created pod ✅
```

---

# ⚙️ 5. How ServiceAccount Works (REAL FLOW)

Inside pod:

```
/var/run/secrets/kubernetes.io/serviceaccount/
```

Contains:

* token (JWT)
* ca.crt
* namespace

---

### Flow:

```
Pod → API Server

1. Pod sends token
2. API validates token
3. RBAC checks permissions
4. Action allowed/denied
```

# 🧪 7. Hands-on Lab (REALISTIC FLOW)

---

# 🧱 Step 0: Setup

```bash
kubectl create ns jenkins
kubectl create sa jenkins-sa -n jenkins
```

---

# 🧪 Lab 1 — Give Jenkins Permissions

---

## 📄 File: `cr-crb.yaml`

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: jenkins-cluster-role

rules:
# Core resources
- apiGroups: [""]
  resources: ["pods", "services", "endpoints"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]

# App resources
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]
```

---

### 🧠 Explanation:

* ClusterRole → reusable + cluster-wide capability
* Gives Jenkins ability to:

  * deploy apps
  * manage workloads

---

## 🔗 Binding

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: jenkins-binding

subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: jenkins
  apiGroup: ""

roleRef:
  kind: ClusterRole
  name: jenkins-cluster-role
  apiGroup: rbac.authorization.k8s.io
```

---

### 🧠 Explanation:

* Binding attaches permission to SA
* `apiGroup: ""` → MUST for ServiceAccount
* ClusterRoleBinding → access across cluster

---

## ▶ Apply:

```bash
kubectl apply -f cr-crb.yaml
```

---

# 🧪 Lab 2 — Token Generation

---

## ❌ Old Way (DON’T USE)

### File: `secret.yaml`

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: jenkins-sa-secret
  namespace: jenkins
  annotations:
    kubernetes.io/service-account.name: jenkins-sa
type: kubernetes.io/service-account-token
```

---

### 🧠 Explanation:

* Creates long-lived token
* No expiry → security risk

---

## ✅ Modern Way (IMPORTANT)

```bash
kubectl create token jenkins-sa -n jenkins
```

---

### 🧠 Explanation:

* short-lived JWT
* recommended
* used in CI/CD

---

# 🔥 8. Pod-Level Demo (MOST ASKED)

---

## 📄 Deployment with ServiceAccount

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: app

  template:
    metadata:
      labels:
        app: app

    spec:
      serviceAccountName: jenkins-sa
      automountServiceAccountToken: true

      containers:
      - name: app
        image: nginx
```

---

### 🧠 Explanation:

* Pod now runs as `jenkins-sa`
* gets token mounted automatically
* can call API server

# 🧠 9. Real Production Flow (IMPORTANT)

---

## 👨‍💼 Admin Setup

```
1. Create ServiceAccount
2. Define RBAC (Role/ClusterRole)
3. Bind permissions
4. Share token mechanism (NOT token itself ideally)
```

---

## 🤖 CI/CD Flow (Jenkins)

```
1. Requests token (TokenRequest API)
2. Uses token
3. Token expires
4. Requests new token
```

---

## 📦 Pod Flow

```
Pod starts → token auto mounted → calls API
```

# 🧠 11. Interview Answers

---

### Q: How do pods authenticate?

👉

“Pods use ServiceAccount tokens automatically mounted inside them, which are validated by the API server.”

---

### Q: Difference between user & ServiceAccount?

👉

“Users are external identities, while ServiceAccounts are Kubernetes-managed identities for workloads.”

---

### Q: How does Jenkins access cluster?

👉

“Using ServiceAccount + RBAC + TokenRequest API to fetch short-lived tokens.”

---

### Q: Why not static tokens?

👉

“They don’t expire and pose major security risks.”

---

### Q: What changed after v1.24?

👉

“ServiceAccount tokens are no longer stored as secrets by default — now short-lived projected tokens are used.”
