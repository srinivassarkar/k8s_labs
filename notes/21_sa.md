# ⚙️ 5. HOW AUTHENTICATION ACTUALLY WORKS (DETAILED)

---

## Inside Pod:

```
/var/run/secrets/kubernetes.io/serviceaccount/
```

Contains:

* token (JWT)
* ca.crt
* namespace

---

## 🔥 FULL FLOW (NOT SIMPLIFIED)

```
1. Pod starts
2. API Server issues token (TokenRequest API internally)
3. Kubelet mounts token
4. Pod sends request with token
5. API Server:
    → verifies signature
    → identifies ServiceAccount
6. RBAC check happens
7. Request allowed/denied
```

---

## 🔐 IMPORTANT DETAIL (MISSED EARLIER)

👉 Token is:

* signed by API server private key
* verified using public key
* contains:

  * subject (SA)
  * namespace
  * expiry
  * audience

# 🔥 6. TOKEN TYPES (COMPLETE + CLEAN)

---

## ✅ 1. Projected Tokens (DEFAULT)

✔ created via TokenRequest API

✔ short-lived (~1 hr)

✔ auto-rotated

✔ mounted in pods

👉 Used by: **internal workloads**

---

## ✅ 2. Manual TokenRequest API

```bash
kubectl create token jenkins-sa
```

✔ short-lived

✔ manual refresh

👉 Used by: **external tools**

---

## ❌ 3. Legacy Secret Tokens

```yaml
type: kubernetes.io/service-account-token
```

❌ long-lived

❌ no rotation

❌ security risk

---

## 🧠 GOLD INSIGHT

```
ALL modern tokens = TokenRequest API
```

Even pod tokens use it internally.

# 🧪 8. HANDS-ON LAB (PRODUCTION STYLE)

---

## Step 1: Create SA

```bash
kubectl create ns jenkins
kubectl create sa jenkins-sa -n jenkins
```

---

## Step 2: RBAC

---

### 📄 `cr-crb.yaml`

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: jenkins-role

rules:
- apiGroups: [""]
  resources: ["pods", "services", "endpoints"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]

- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]
```

---

### 🔗 Binding

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: jenkins-binding

subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: jenkins
  apiGroup: ""

roleRef:
  kind: ClusterRole
  name: jenkins-role
  apiGroup: rbac.authorization.k8s.io
```

---

## Step 3: Token (REAL WAY)

```bash
kubectl create token jenkins-sa -n jenkins
```

---

## Step 4: Use

```bash
curl -H "Authorization: Bearer <TOKEN>" https://<api-server>/api -k
```

# 🧠 11. INTERVIEW KILLER ANSWERS

---

### Q: How do pods authenticate?

👉

“Using projected ServiceAccount tokens mounted automatically, validated by API server.”

---

### Q: Internal vs external access?

👉

“Internal workloads use projected tokens, external systems use TokenRequest API.”

---

### Q: Why ServiceAccounts?

👉

“To separate machine identity from human users and enable fine-grained RBAC.”

---

### Q: What changed after v1.24?

👉

“Tokens are no longer stored as Secrets by default — now short-lived projected tokens are used.”
