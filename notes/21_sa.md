# Kubernetes Authentication & Service Accounts
### The Complete Mental Model — Interviews · CKA · Production

---

## 0. First Principles — The Mental Model That Never Changes

Before anything else, lock this in:

```
AUTHENTICATION  →  WHO are you?
AUTHORIZATION   →  WHAT can you do?
```

Every interaction with Kubernetes collapses into this single flow:

```
Request → API Server → AUTH → AUTHZ → EXECUTE
```

**The API Server is always the gatekeeper. No exceptions.**

Nothing bypasses it. No request gets executed without passing through authentication and then authorization. If you internalize only one thing from this document, it's this.

---

## 1. Reality Constraints — What Kubernetes Actually Does and Doesn't Do

This is the single most important conceptual correction:

| What Kubernetes Does | What Kubernetes Doesn't Do |
|---|---|
| Authenticates requests | Manage human user accounts |
| Authorizes via RBAC | Store users in a database |
| Issues tokens for workloads | Act as an identity provider for humans |

```
Humans    →  Identity comes from OUTSIDE Kubernetes (certs, OIDC, LDAP)
Machines  →  Identity comes from ServiceAccounts (managed by Kubernetes)
```

There is no `kubectl create user`. There is no user object in etcd. Human identity is always delegated to an external system. Machine identity — pods, CI tools, controllers — is managed natively via ServiceAccounts.

## 2. The Complete Auth Landscape — All Methods, All Actors

### Who Talks to the Cluster?

```
┌─────────────────────────────────────────────────────┐
│                  CLUSTER CLIENTS                    │
│                                                     │
│  HUMANS               │  MACHINES                  │
│  ─────────────────    │  ─────────────────────────  │
│  kubectl              │  INTERNAL (in-cluster)      │
│  Dashboard            │    Prometheus               │
│  CI/CD operators      │    ArgoCD                   │
│                       │    Fluentd / Falco           │
│                       │                             │
│                       │  EXTERNAL (out-of-cluster)  │
│                       │    Jenkins                  │
│                       │    GitHub Actions           │
│                       │    Terraform                │
└─────────────────────────────────────────────────────┘
```

### Auth Methods by Actor

| Method | Used By | Production Status | Notes |
|---|---|---|---|
| Basic Auth | Humans | ❌ Dead | Never use |
| Static Token File | Humans | ❌ Dead | Never use |
| X.509 Certificates | Humans | ✅ Very common | `kubeconfig` based |
| OIDC / LDAP | Humans | ✅ Production standard | e.g. Google, Okta |
| Projected SA Tokens | Machines (internal) | ✅ Default | Auto-mounted in pods |
| TokenRequest API | Machines (external) | ✅ Modern standard | Short-lived, on-demand |
| Legacy Secret Tokens | Machines (any) | ❌ Deprecated | Long-lived, security risk |

### Key Insight

```
Kubernetes performs authentication.
But for humans, identity is always sourced from outside.
For machines, Kubernetes is the identity provider via ServiceAccounts.
```

## 3. Decision Logic — When to Use What

Use this flowchart every time you design access:

```
Who needs access?
       │
       ├─── Human
       │         └── Use certificates (admin/dev) or OIDC (org-level SSO)
       │
       └─── Machine
                 │
                 ├─── Lives INSIDE the cluster?
                 │         └── ServiceAccount + projected token (auto-mounted)
                 │
                 └─── Lives OUTSIDE the cluster?
                           └── TokenRequest API (kubectl create token ...)
```

### Token Type Decision

| Situation | Token Type | Why |
|---|---|---|
| Pod needs API access | Projected token | Auto-mounted, auto-rotated, scoped |
| Jenkins/GH Actions needs API access | TokenRequest API | Short-lived, manually refreshed |
| Legacy system, pre-v1.24 | Secret-based token | Only for backward compat — avoid |
| High-security orgs | OIDC + Workload Identity | No tokens stored anywhere |

> **Never** give a pod or CI system `cluster-admin`. Never use the `default` ServiceAccount for anything with real permissions. Never use long-lived tokens when short-lived ones are available.

## 4. Internal Working — How Authentication Actually Happens

### Inside a Pod

When a pod starts, Kubernetes automatically mounts a token at:

```
/var/run/secrets/kubernetes.io/serviceaccount/
├── token      ← JWT (the credential)
├── ca.crt     ← CA to verify the API server
└── namespace  ← namespace the pod runs in
```

### The Full Authentication Flow (Step by Step)

```
1. Pod is scheduled and starts
2. API Server calls TokenRequest API internally
3. Kubelet receives and mounts the projected token into the pod
4. Pod makes a request to API Server with:
   Authorization: Bearer <token>
5. API Server:
   → Verifies the token signature using its public key
   → Decodes the JWT payload
   → Identifies the ServiceAccount (subject)
6. RBAC check:
   → Looks up RoleBindings/ClusterRoleBindings for that SA
   → Checks if the requested verb+resource is allowed
7. Request is ALLOWED or DENIED
```

### What's Inside the JWT Token

```json
{
  "iss": "kubernetes/serviceaccount",
  "sub": "system:serviceaccount:jenkins:jenkins-sa",
  "aud": ["kubernetes"],
  "exp": 1700000000,
  "namespace": "jenkins",
  "serviceaccount": {
    "name": "jenkins-sa",
    "uid": "..."
  }
}
```

Key fields:
- **sub** — the ServiceAccount identity (what RBAC checks against)
- **aud** — audience binding (prevents token reuse across systems)
- **exp** — expiry (short-lived by design)

### ServiceAccount Groups (Often Missed)

When a SA authenticates, it belongs to **groups automatically**:

```
system:serviceaccount:<namespace>:<name>  ← specific SA
system:serviceaccounts:<namespace>        ← all SAs in namespace
system:serviceaccounts                    ← all SAs in cluster
```

You can bind RBAC to an entire namespace's SAs at once:

```yaml
subjects:
- kind: Group
  name: system:serviceaccounts:jenkins
  apiGroup: rbac.authorization.k8s.io
```

This grants access to **every** ServiceAccount in the `jenkins` namespace — use carefully.

### Token Rotation Reality

| Token Type | Rotation |
|---|---|
| Projected tokens (pod) | ✅ Automatic — kubelet handles it |
| TokenRequest API (external) | ❌ Manual — you must refresh |
| Legacy Secret tokens | ❌ Never rotates — static forever |

> **Interview trap:** "Do tokens auto-rotate?" → Only projected pod tokens. External tokens are your responsibility.

## 5. Hands-On — YAML + Commands

### Step 1 — Create the Namespace and ServiceAccount

```bash
kubectl create ns jenkins
kubectl create sa jenkins-sa -n jenkins
```

### Step 2 — RBAC: ClusterRole

```yaml
# cr-crb.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: jenkins-role
rules:
- apiGroups: [""]
  resources: ["pods", "services", "endpoints"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "delete", "patch", "update"]
```

### Step 3 — RBAC: ClusterRoleBinding

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: jenkins-binding
subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: jenkins
  apiGroup: ""
roleRef:
  kind: ClusterRole
  name: jenkins-role
  apiGroup: rbac.authorization.k8s.io
```

```bash
kubectl apply -f cr-crb.yaml
```

### Step 4 — Generate a Token (Modern Way)

```bash
# Short-lived token for external use
kubectl create token jenkins-sa -n jenkins

# With custom expiry
kubectl create token jenkins-sa -n jenkins --duration=1h
```

### Step 5 — Use the Token

```bash
TOKEN=$(kubectl create token jenkins-sa -n jenkins)
curl -H "Authorization: Bearer $TOKEN" https://<api-server>/api -k
```

### Step 6 — Assign SA to a Pod

```yaml
spec:
  serviceAccountName: jenkins-sa
  # token is auto-mounted — no manual steps needed
```

## 5b. Legacy Secret-Based Tokens — Know It, Don't Use It

### The Old Flow (pre-v1.24)

```
1. Create ServiceAccount
2. Create a Secret with annotation pointing to the SA
3. Kubernetes generates a token and stores it in the Secret (in etcd)
4. Extract token manually
5. Token never expires ← the problem
```

### YAML You Must Recognize in Interviews

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: jenkins-sa-secret
  namespace: jenkins
  annotations:
    kubernetes.io/service-account.name: jenkins-sa   # ← critical annotation
type: kubernetes.io/service-account-token
```

```bash
kubectl apply -f secret.yaml

# Extract the token
kubectl get secret jenkins-sa-secret -n jenkins \
  -o jsonpath="{.data.token}" | base64 --decode
```

### Why This Was Changed in v1.24

| Before v1.24 | After v1.24 |
|---|---|
| ServiceAccount → auto-creates Secret | No auto-created Secret |
| Token stored in etcd | Token issued on demand via TokenRequest |
| Token never expires | Token has expiry (`exp` claim) |
| Rotation = manual | Rotation = automatic (for pods) |

### Token Type Comparison

```
Secret Token        →  static, long-lived, stored in etcd  ❌ deprecated
Projected Token     →  auto-mounted, short-lived, auto-rotated  ✅ default
TokenRequest API    →  on-demand, short-lived, for external tools  ✅ modern
```

### When Would You Ever Use Legacy Tokens?

Only in these narrow cases:
- **Legacy clusters** running Kubernetes < v1.24
- **Backward compatibility** with systems that cannot refresh tokens
- **Air-gapped quick hacks** — still risky, still not recommended

> **Interview positioning:** "I would not use Secret-based tokens. They don't expire and are stored in etcd, which creates a permanent security exposure. The TokenRequest API is the right approach."

## 6. Production Flow — Real-World Architecture

### Internal Tools: The Agent Model

Tools like Prometheus, ArgoCD, and Fluentd follow the **agent model**:

```
┌────────────────────────────────────────────┐
│              CLUSTER                       │
│                                            │
│  [Prometheus Agent] ←→ [API Server]        │
│       ↑ (projected token, auto-rotated)    │
│                                            │
└────────────────────────────────────────────┘
         ↑
  [External Grafana UI] talks to Prometheus agent
  NOT to the API Server directly
```

The external UI never touches the API Server. The agent inside the cluster does, using its ServiceAccount token. This is why almost every production monitoring, GitOps, and logging tool follows the same pattern.

### External Tools: The TokenRequest Flow

Tools like Jenkins and GitHub Actions that live outside the cluster:

```
1. Jenkins requests a token → kubectl create token jenkins-sa
2. API Server issues short-lived token
3. Jenkins calls API Server with token
4. Token expires after TTL
5. Jenkins must request a new token
```

### Gold Standard (Advanced Organizations)

```
OIDC + Workload Identity
→ No tokens stored anywhere
→ Identity federated from the CI system's OIDC issuer (e.g. GitHub Actions OIDC)
→ Kubernetes trusts the external OIDC issuer directly
```

### The `automountServiceAccountToken` Control

By default, every pod gets a token mounted. For pods that don't need API access, disable it:

```yaml
# On the ServiceAccount (applies to all pods using it)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
automountServiceAccountToken: false

# Or on the Pod spec (overrides SA setting)
spec:
  automountServiceAccountToken: false
```

> **Why it matters:** Reduces blast radius if the pod is compromised. If your app doesn't call the Kubernetes API, it shouldn't have credentials that can.

### The Audience (`aud`) Claim

Modern tokens are **audience-bound**:

```json
"aud": ["kubernetes"]
```

A token issued for `kubernetes` cannot be replayed against a different system (e.g., your internal metrics API). This prevents stolen tokens from being misused across services.

You can issue tokens with custom audiences:

```bash
kubectl create token jenkins-sa --audience=https://my-internal-service
```

## 7. Mistakes — What Actually Breaks in Real Life

### Mistake 1 — Giving `cluster-admin` to Automation

```yaml
# ❌ This is what gets companies breached
roleRef:
  kind: ClusterRole
  name: cluster-admin  # ← NEVER for CI/CD
```

Automation should have the minimum permissions it needs. Define a custom ClusterRole with exactly the verbs and resources required.

### Mistake 2 — Using the `default` ServiceAccount

Every namespace has a `default` SA. Pods use it automatically if you don't specify one. If anyone binds permissions to `default`, **every pod in the namespace** gets those permissions.

```bash
# ❌ This pattern is dangerous
kubectl create rolebinding ... --serviceaccount=myns:default
```

Always create a dedicated SA for each workload.

### Mistake 3 — Long-Lived Tokens in CI/CD

```
Jenkins hardcodes token in pipeline config
Token leaks via log, screenshot, or config repo
Attacker has permanent cluster access
```

Fix: Use `kubectl create token` with short TTL. Rotate before it expires. Use OIDC federation if possible.

### Mistake 4 — Token Expires Mid-Pipeline

```
Jenkins pipeline starts with valid token
Pipeline runs > 1 hour
Token expires
API calls start returning 401
Pipeline fails
```

Fix: Request a token with sufficient TTL at the start of the pipeline, or implement token refresh logic.

### Mistake 5 — Pod Gets 403

Common causes:
1. Pod is using `default` SA which has no permissions
2. Forgot to create the RoleBinding
3. RoleBinding is in wrong namespace (Role vs ClusterRole confusion)
4. `serviceAccountName` typo in pod spec

### Mistake 6 — Namespace Mismatch in Binding

```yaml
subjects:
- kind: ServiceAccount
  name: jenkins-sa
  namespace: production  # ← must match where the SA actually lives
```

The namespace in `subjects` must exactly match the namespace where the ServiceAccount was created.

### Mistake 7 — Role vs ClusterRole Confusion

| Resource | Scope |
|---|---|
| Role | Single namespace |
| ClusterRole | Cluster-wide OR reusable across namespaces |
| RoleBinding | Grants within a namespace |
| ClusterRoleBinding | Grants cluster-wide |

You **can** bind a ClusterRole with a RoleBinding — it restricts the permissions to that namespace. You **cannot** bind a namespace-scoped Role with a ClusterRoleBinding.

## 8. Interview Answers — Compressed Articulation

### Q: How do pods authenticate to the Kubernetes API?

> "Pods use projected ServiceAccount tokens that are automatically mounted by the kubelet at `/var/run/secrets/kubernetes.io/serviceaccount/token`. These tokens are short-lived JWTs issued via the TokenRequest API. The API Server validates the token signature using its private key and then runs an RBAC check against the ServiceAccount's bound roles."

### Q: What's the difference between how internal and external tools authenticate?

> "Internal workloads like Prometheus or ArgoCD use projected ServiceAccount tokens that are auto-mounted and auto-rotated by the kubelet — they never expire unexpectedly. External tools like Jenkins use the TokenRequest API to obtain short-lived tokens on demand, which they must refresh manually before expiry."

### Q: Why use ServiceAccounts instead of user credentials for automation?

> "ServiceAccounts give machines a distinct, auditable identity separate from human users. This means your audit log shows `system:serviceaccount:jenkins:jenkins-sa` performed a deployment — not `admin`. It also enables fine-grained RBAC: each workload gets exactly the permissions it needs, nothing more."

### Q: What changed in Kubernetes v1.24 regarding tokens?

> "Before v1.24, every ServiceAccount automatically got a Secret containing a long-lived token with no expiry. After v1.24, that automatic Secret creation was removed. Tokens are now issued on demand via the TokenRequest API, they carry an expiry claim, and projected tokens inside pods are automatically rotated by the kubelet."

### Q: Would you ever use Secret-based long-lived tokens?

> "No. They don't expire, they're stored in etcd, and rotation is entirely manual — which means it almost never happens. The correct approach is the TokenRequest API for external tools and projected tokens for pods. For advanced setups, OIDC federation eliminates the need to manage tokens altogether."

### Q: What is `automountServiceAccountToken` and when do you use it?

> "It's a field on the ServiceAccount or Pod spec that controls whether a token is automatically mounted. I disable it for any pod that doesn't need to call the Kubernetes API. It reduces attack surface — a compromised pod with no token mounted can't pivot to the API Server."

### Q: A pod is getting 403 errors calling the API. How do you debug it?

> "I'd start by checking if the pod has the right `serviceAccountName` set, then verify the ServiceAccount exists in the correct namespace. Next I'd check if there's a RoleBinding or ClusterRoleBinding linking that SA to a role that covers the requested resource and verb. I'd use `kubectl auth can-i` to test the exact permission."

## 9. Debugging — How to Fix Issues Fast

### Check What a ServiceAccount Can Do

```bash
# Can jenkins-sa get pods in the jenkins namespace?
kubectl auth can-i get pods \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins

# Can it create deployments?
kubectl auth can-i create deployments \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins

# Full permission dump
kubectl auth can-i --list \
  --as=system:serviceaccount:jenkins:jenkins-sa \
  -n jenkins
```

### Check What's Mounted in a Pod

```bash
kubectl describe pod <pod-name>
# Look for: "Mounts:" section — is the SA token listed?

# Check which SA the pod is using
kubectl get pod <pod-name> -o jsonpath='{.spec.serviceAccountName}'
```

### Inspect the Token Itself

```bash
# Get the token from inside a running pod
kubectl exec <pod-name> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token

# Decode it (paste to jwt.io or use jq)
TOKEN=$(kubectl exec <pod> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token)
echo $TOKEN | cut -d. -f2 | base64 -d | jq .
```

### Check RBAC Bindings

```bash
# List all RoleBindings in a namespace
kubectl get rolebindings -n jenkins -o wide

# List ClusterRoleBindings referencing a SA
kubectl get clusterrolebindings -o json | \
  jq '.items[] | select(.subjects[]?.name=="jenkins-sa")'

# Describe a specific binding
kubectl describe clusterrolebinding jenkins-binding
```

### List All ServiceAccounts

```bash
kubectl get sa -A                         # all namespaces
kubectl get sa -n jenkins                 # specific namespace
kubectl describe sa jenkins-sa -n jenkins # full details
```

### Common 403 Diagnosis Path

```
403 on API call
   ↓
kubectl auth can-i [verb] [resource] --as=system:serviceaccount:[ns]:[sa]
   ↓
If "no" → check RoleBinding exists and is in right namespace
   ↓
kubectl get rolebindings,clusterrolebindings -A | grep [sa-name]
   ↓
If binding exists → check Role has right verbs/resources/apiGroups
   ↓
kubectl describe clusterrole [role-name]
```

## 10. Kill Switch — 10-Second Recall

```
WHO     →  Humans (external identity) vs Machines (ServiceAccount)
HOW     →  Certs/OIDC for humans | Projected token for pods | TokenRequest for external
WHERE   →  RBAC decides what's allowed — SA alone gives nothing

Internal  →  projected token (auto-mounted, auto-rotated by kubelet)
External  →  TokenRequest API (manual refresh required)
Legacy    →  Secret token (no expiry, never rotate, avoid always)

Token = identity proof
RBAC  = permission check
API Server = always the authority, never bypassed

v1.24+: No auto-Secret. No long-lived tokens by default.
automountServiceAccountToken: false → reduce blast radius
default SA → never bind permissions to it
cluster-admin → never for automation
```
